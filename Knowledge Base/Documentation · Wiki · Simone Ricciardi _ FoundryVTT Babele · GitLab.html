    Babele: Translation of Foundry VTT Content

    Babele is a module for Foundry Virtual Tabletop (VTT) that allows translating the content of compendiums. This module is designed to enhance the user experience by making data available in the desired language without altering the original compendium files.

    Note: this documentation always refers to the latest version of the module.

    Introduction

    Babele is designed to meet the needs of game masters and players who wish to use Foundry VTT in their preferred language. With this plug-in, you can:

    • Automatically translate compendiums with customized text strings.
    • Keep translations separate from the original compendium files, ensuring a non-invasive approach.
    • Support multiple languages for various projects.

    This module is meant to be flexible and easy to configure.

    Requirements

    For Translators

    If you plan to create and distribute translations as a module, the following are required:

    • Foundry Virtual Tabletop version 0.7.5 or later.
    • Basic knowledge on how to create and distribute a Foundry module (see here)
    • A text editor to configure JSON files, such as Visual Studio Code or Notepad++.

    For End Users

    If you simply want to use translations provided by an existing module:

    • Foundry Virtual Tabletop installed.
    • The Babele module installed and enabled.
    • The translation module that contains the translations you want to use.

    Installation

    Steps for Installation

    1. Add the Module to Foundry VTT
      • Search and add the module via Foundry’s "Module Manager."
      • Alternatively, download the latest version of the module from the Babele GitLab repository: https://gitlab.com/riccisi/foundryvtt-babele and copy the module folder into Foundry VTT’s modules directory.
    2. Enable the Module
      • Go to the world settings in Foundry.
      • Select the "Babele" module in the "Enabled Modules" section and confirm.
    3. Set your language
      • Go to "Configure Settings" and change the "Language Preference" to that relating to translation.

    Configuration

    Creating Translation Files

    For every compendium you want to translate, you need to create a dedicated file in JSON format. These files will contain the translations for the entities in the compendium.

    Structure of the JSON File:

    {
  "label": "compendium_label",
  "folders": {
    "Folder A": "folder_A_translation",
    "Folder B": "folder_B_translation",
    "Folder C": "folder_C_translation"
  },
  "entries": {
    "document_name": {
      "field": "field_translation"
    }       
  }
}

    Field Descriptions:

    • label: The label (title) of the compendium or system to which the compendium belongs.
    • folders: (If the compendium contains folders) An object containing translations, where each key is the folder name in the compendium, and the value is translations.
    • entries: An object containing translations, where each key is the entity Name/ID in the compendium, and the value is another object with specific field translations.

    Note: Babele also accepts another format for the entries, call it 'legacy', with an array instead of an object, like this:

    "entries": [{
  "id": "document_name"
  "field": "field_translation"
}]

    This format was used in the first versions and is still supported, but is no longer considered as a 'official' format, as the object format is more user friendly and offers greater compatibility with translation tools such as Webplate or Transfex.

    Example:

    {
  "label": "Incantesimi (SRD)",
  "folders": {
    "1st Level": "1° Livello"
  }
  "entries": {
    "Fireball": {
      "name": "Palla di Fuoco",
      "description": "Una brillante esplosione di fuoco che infligge danni in una determinata area."
    },
    "Magic Missile": {
      "name": "Dardo Incantato",
      "description": "Lancia dardi magici che colpiscono automaticamente i bersagli."
    }
  }
}

    In this example, we are translating in Italian two spells from the "dnd5e" system and a folder. The entity and the folder NAME (e.g., "Fireball") must match the NAME used in the original compendium.

    note: It is also possible to specify the DOCUMENT ID instead of the NAME (which can be obtained directly from the Document Sheet). Even if in this way the file certainly becomes less user friendly, the use of the ID becomes important in cases of entry with the same name or in general to be sure that Babele applies the translation correctly regardless of the name (which could also change).

    As for the "folders" the format is pretty simple: original name in key and translation in the value. No need to manage the depth, if a name is used multiple times in the compendium, only one translation is necessary.

    Translate "level 0" compendium folders

    Systems and modules can place compendiums inside folders.
    This is not the same as the folders that are present in the compendiums.

    In order to translate these folders, specials translations files are used. They should be named [id]._packs-folders.json, where [id] must be replaced by the ID of the concerned system or the module ( eg. pf1._packs-folders.json).

    As with others translations files, the file should contains an entries key, with translations in it:

    {
  "entries": {
    "Character Building": "Création de personnages",
    "Roll Tables": "Tables aléatoires"
  }
}

    Mapping Entity Types to Translate

    Babele uses a mapping to determine which entity fields need translation. By default, the generic fields "name" and "description" are considered for translation. Further fields are then considered depending on the type of the document (actor, item, journal, etc ...).

    Babele is an 'agnostic' or 'system independent' module, consequently the fields that are mapped for the various types of document are only the most generic ones and independent of the game system as much as possible.
    However, this mapping can be extended or overridden as needed to include other system/module-specific fields.

    See here the default mappings that Babele applies based on the document type.

    Defining a Custom Mapping:

    Based on the game system, the mapping of all various types of document can be enriched with further fields that possibly need to be translated. To extend or override the default mapping applied by Babele it is possible to define a custom mapping directly within the compendium translation file or in a generic external separate JSON file.

    It is also possible to provide a custom mapping in case of Adventure Translation or Embedded Documents, see the specific cases in the Advanced Mappings section.

    Mapping definition - within the compendium translation file.

    The mapping defined within the compendium translation file has a structure like the following:

    {
  "mapping": {
     "translated_field": "original_field"
  }
}

    Field Descriptions:

    • mapping: Upper level properties containing an object with the definition of the mapping.
    • translated_field: The field name in each entry containing the relative translation.
    • original_field: The complete path to the field in the original document data.

    Mapping Example:

    {
  "label": "Incantesimi (SRD)",
  "mapping": {
    "description": "data.description",
    "materials": "data.materials.value"
  },
  "entries": {
    "Haste": {
      "name": "Velocità",
      "description": "<p>L'incantatore...",
      "materials": "Una piccola striscia di stoffa bianca."
    }
  } 
}

    In this example, we define a custom mapping, which will be used only for the associated compendium (so to a specific document Type), specifying that for all contained entities the "data.description.value" field should be translated using the "description" field in the entry (overriding the default definition) and the "data.materials.value" field should be translated as "materials". (an added specific field).

    Mapping definition - external separate JSON file:

    If you want to extend or overwrite the default mapping that Babele applies to each document type in a single place, you can create a JSON file by denominating it "mappings.json", putting the definitions by type inside, using the following structure:

    {
  "document_type": {
     "translated_field": "original_field"
  }
}

    Field Descriptions:

    • document_type: The type of entity in the compendium (e.g., "Item", "Actor").
    • translated_field: The field name in each entry containing the relative translation.
    • original_field: The complete path to the field in the original document data.

    Mapping Example:

    {
  "Item": {
    "description": "data.description.value",
    "source": "data.source"
  },
  "Actor": {
    "bio": "data.biography.value"
  }
}

    In this way, the default mapping is extended with these further definitions and will be applied regardless of the compendium, obviously respecting the type of the contained elements.

    Placement of Translation Files

    To make the translation files available to Babele, you need to respect a naming convention and a directory structure. In addition, this procedure varies according to whether you want the translations to be only local to your Foundry installation or if you want to be distributed to be used by others.

    Simple local translations:

    If your purpose is to translate your world's compendiums just for you and your players and you don't need to distribute them to others or use advanced mapping features, then the easiest way is to provide by a local directory.

    Create a directory under User Data and specify the path in 'Game Settings'> 'Babele'> 'Translation Files Directory'. This directory will be used as a 'root' directory for translation files.

    Since Babele applies the translation files according to the language selected in the settings, create a subdirectory per language and place the translation files inside, for example:

    user_data/
  translations/
      it/
        compendium_name.json

    The name of the translation file MUST be by the name of the associated collection (ex. dnd5e.spells.json). One file per pack is required.

    Ensure that the folder structure and file names are correct to guarantee that Babele can load the appropriate translations.

    Distributing Translations in a Module:

    If your aim is to distribute translations to others or to be able to use them on different Foundry instances then the most effective way is to include them in a module.

    First, therefore, you will need to create and make a module available to Foundry. Follow the instructions in the official documentation for creation and configuration. It is also possible to create a new module using the procedure exposed in the Foundry setup directly from the UI. After that, register the module using the Babele’s API in your included script file (preferably esmodules) so that it can be used as a source of translations, below is an example:

    Hooks.once('babele.init', (babele) => {
    babele.register({
      module: 'FoundryVTT-dnd5e-it',
      lang: 'it',
      dir: 'compendium'
    });
});

    Parameter Descriptions

    • module: The name of the module registering the translations.
    • lang: The language of the translations (e.g., "it" for Italian).
    • dir: The path to the directory containing the translation files.

    Folder Structure

    Ensure that the directory specified in the dir parameter contains the all your JSON translation files. Example:

    modules/
  your_module_name/
    module_dir/
        compendium_name.json

    Distributing Translations in a System:

    It is also possible for system developers to distribute translations made with Babele together with the distribution of their own system. It is sufficient to use the following API to provide Babele with the system directory in which to find, divided into folders one for each language, the JSON files of the translated compendia:

    Hooks.once('babele.init', (babele) => {
    babele.setSystemTranslationsDir("packs/translations");
});

    Folder Structure

    systems/
  your_system_name/
     packs/
        translations/
            /it
            /fr

    Placement of Mapping File

    In case you want to use a single external mapping File File to enrich/overwrite Babele's default mapping, this file must be provided in one of these ways:

    Local:

    Place your mapping file under the directory path configured in 'Game Settings'> 'Babele'> 'Translation Files Directory', example:

    user_data/
  translations/
        mapping.json

    note: It must not be put under the subdirectory of the language, as it is supposed that the mapping can be used independently of the language

    Module:

    Place your mapping file under the module directory you provided in the registration, example:

    modules/
  your_module_name/
    module_dir/
        mapping.json

    System:

    Place your mapping file under the system translation dir you provided in the setSystemTranslationsDir API call, example:

    systems/
  your_system_name/
     packs/
        translations/
           mapping.json

    Advanced Mappings

    Using a module to distribute translations, you can to take advantage of advanced mapping techniques, such as the use of Converters.

    Using Converters for Dynamic Translation

    Babele supports the use of converters, which are custom JavaScript functions designed to handle complex or dynamic translations. Converters can be registered and applied to specific fields, allowing for on-the-fly processing of data.

    Defining a Converter:

    A converter is a JavaScript function that takes the original value of a field and returns the translated value. For example:

    function myConverter(originalValue) {
  return `Translated: ${originalValue}`;
}

    Registering a Converter:

    To register a converter, use the Babele.registerConverter method:

    babele.registerConverter('capitalize', (originalValue) => {
  return originalValue.toUpperCase();
});

    Using Converters in Mapping:

    Once registered, a converter can be referenced in the mapping:

    {
  "mapping": {
    "description": {
      "path": "data.description.value",
      "converter": "capitalize"
    }
  }
}

    In this example, the "capitalize" function will be applied to the data.description.value field of each entities during translation.

    Complete Converter Signature

    The converter is invoked by passing further arguments in addition to the value to be translated:

    function converter(originalValue, translation, data, translatedCompendium, allTranslations)

    Args Descriptions:

    • originalValue: value relative to the path within the document data to be translated.
    • translation: 'static' translation referred to the mapping field inside the entry, if present.
    • data: untranslated data of the current document
    • translatedCompendium: current TranslatedCompendium object which can be used to dynamically translate other fields of the same document or other belonging to the same compendium.
    • allTranslations:

    A typical use case of a converter can be, for example, to convert one unit of measurement into another:

    babele.registerConverter('feetToMeters', (ft) => {
  if(!ft) {
    return ft;
  }
  return round(parseInt(ft)*0.3);
});

    Or replace a label from a default domain list:

    const rarity = {
  "Common": "Comune",
  "Uncommon": "Non Comune",
  "Rare": "Raro",
  "Very rare": "Molto Raro",
  "Legendary": "Leggendario"
};

babele.registerConverter('rarity', (r) => {
  return rarity[r] ? rarity[r] : r
});

    These are simple cases, but a converter can be used to manage much more complex cases, such as translating entities within entities or seeking translations in other compendiums, etc. At low level, for Babele, each translation of each field is transparently managed through a converter, making it modular and extendable.

    Included converters

    Babele himself uses specific converters for the translation of complex elements within the documents, Below is the list of Converters which are already included and can be used for your own needs.

    fromPack

    The converter fromPack is used to get translation from another pack, by default Item.

    Actor mapping allows automatic translation of owned items based on translations already present in the other compendiums, as shown below:

    {
    "Actor": {
        "name": "name",
        "description": "system.details.biography.value",
        "items": {
            "path": "items",
            "converter": "fromPack"
        },
        "tokenName": {
            "path": "prototypeToken.name",
            "converter": "name"
        }
    }
}

    The fromPack converter also allows you to overwrite any item if the translation from its translated compendium is not adequate or if some items are present and others are not. To overwrite (or add) owned items translations, simply provide them as under translation under "items":

    {
    "label": "Mostri (SRD)",
    "entries": [
        {
            "id": "Acolyte",
            "name": "Accolito",
            "description": "...",
            "items": [
                {
                    "id": "Club",
                    "name": "Randello dell'accolito",
                    "description": "Il randello dell'accolito è un'arma molto potente..."
                }
            ]
        }
    ]
}

    In the example above, the "Club" item will be overwritten with the provided custom translation while the one from the item compendium will be skipped.

    The other items not overwritten will be translated by taking the translations from the relative compendium (if any).
    Of course, the transifex-compatible format is also supported:

    {
    "label": "Mostri (SRD)",
    "entries": {
        "Acolyte": {
            "name": "Accolito",
            "description": "...",
            "items": {
                "Club": {
                    "name": "Randello dell'accolito",
                    "description": "Il randello dell'accolito è un'arma molto potente..."
                }
            }
        }
    }
}

    How Babele Works

    Once configured, Babele intercepts compendium loading requests and replaces the original strings with the translations specified in the JSON files. This happens in real-time and does not alter the original files.

    Key Features:

    • Dynamic translation based on the selected language.
    • Support for nested keys and complex strings.
